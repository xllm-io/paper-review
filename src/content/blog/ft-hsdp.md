## 二、核心问题

### 2.1 大规模训练的效率困境

在 100K GPU 规模下，同步训练面临严重效率问题：

| 指标 | 数据 |
|------|------|
| **故障频率** | 每 18 分钟一次（100K GPU） |
| **同步恢复时间** | ~10 分钟 |
| **有效训练时间** | 仅 44% (8/18 分钟) |

### 2.2 故障类型分布（32K GPU 实验数据）

| 故障类型 | 占比 |
|----------|------|
| GPU HBM3 内存 | 22.9% |
| PCIe 设备 | 18.0% |
| NCCL 看门狗超时 | 9.0% |
| GPU 计算错误 | 7.4% |
| 软件 Bug | 7.1% |
| 主机维护 | 6.2% |
| 内核故障 | 5.8% |
| 系统重启 | 5.6% |
| 数值/静默数据损坏 | 5.5% |
| 网络交换机/线缆 | 5.3% |
| SSD | 4.4% |

**关键发现**：
- 78% 中断由硬件故障引起
- HBM 问题已成为首要故障源
- 数值问题（NaN/SDC）调试困难，需要数小时到数天

## 三、核心思想：FT-HSDP

### 3.1 设计理念

**Fault Tolerant Hybrid-Shared Data Parallelism (FT-HSDP)** 的核心思想：

1. **数据并行副本作为容错单元**：创建多个副本（replica），每个副本独立训练
2. **故障隔离**：故障发生时，仅下线包含故障 GPU 的单个副本
3. **异步恢复**：故障副本恢复期间，其他副本继续训练

### 3.2 架构设计（Figure 3）

```
┌─────────────────────────────────────────────────────────┐
│                    FT-HSDP 架构                          │
├─────────────────────────────────────────────────────────┤
│  Replica 0    Replica 1    Replica 2    Replica 3      │
│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐         │
│  │8192  │    │8192  │    │8192  │    │8192  │         │
│  │GPU   │    │GPU   │    │GPU   │    │GPU   │         │
│  └──┬───┘    └──┬───┘    └──┬───┘    └──┬───┘         │
│     │           │           │           │              │
│     └───────────┴───────────┴───────────┘              │
│                    FTAR 梯度交换                         │
└─────────────────────────────────────────────────────────┘
```

### 3.3 关键技术

#### 技术 1：Fault Tolerant All Reduce (FTAR)（Figure 5）

**设计目标**：
- 网络感知架构（跨 DC 通信）
- 可重建通信组
- 简单故障管理
- 最小化与计算内核的资源争用

**CPU-GPU 混合设计**：
- **CPU 负责控制平面**：连接管理、故障检测、拥塞控制
- **GPU 负责数据平面**：实际数据传输，最大化性能

**Ring 算法**：
- 限制每个 GPU 仅与两个邻居通信
- 最小化并发网络流量
- ReduceScatter + AllGather 两阶段

**性能优化**：
- 固定大小块流水线
- 指令级并行优化
- 仅使用 2 个 SM（NCCL 需要 4 个）
- 8MB 块大小，完全隐藏归约开销

#### 技术 2：Non-blocking Catch Up（Figure 6）

**核心创新**：利用训练系统的特殊属性实现非阻塞追赶

**传统方法的问题**：
- 落后副本需要追赶新任务
- 通常需要阻塞或减慢其他副本

**FT-HSDP 方案**：

```
步骤 n 开始：
  - 健康副本：报告下一步号 n
  - 落后副本：报告步骤号 n-4

步骤 n 结束：
  - 健康副本：发送真实梯度
  - 落后副本：发送零梯度

结果：所有副本同步到相同状态
```

**关键洞察**：
1. 发送零梯度即可达到与执行训练相同的状态
2. 检查点获取通常比训练一步更快

#### 技术 3：高效检查点与恢复（Figure 7）

**检查点策略**：
- 每 100 步写入持久存储
- 模型状态和优化器状态仅一个副本写入
- DataLoader 状态每步写入（很小）

**恢复流程**：
- 从健康副本直接获取状态（而非持久存储）
- 负载均衡组织获取（图 7）
- 通过 HTTP 协议（无 GPU 高速网络争用）

### 4.2 延迟特性

| 跳数 | 延迟倍数 |
|------|----------|
| 同 Rack | 1x（基准） |
| 不同 Rack，同 AI Zone | 7x |
| 不同 AI Zone | 15x |
| 不同 DC Building | 30x |

**设计原则**：
- 同一副本内 GPU 放在同一 DC（低延迟）
- 不同副本放在不同 DC（容忍高延迟）

## 五、实验结果

### 5.1 实验配置（Figure 2, 8, 9）

| 项目 | 配置 |
|------|------|
| **GPU** | 98K H100 |
| **副本数** | 12 |
| **每副本 GPU** | 8192 |
| **DC 数量** | 4 |
| **模型** | 数万亿参数稠密 Transformer |
| **并行策略** | FSDP + 异步 TP + CP + 自定义 PP |

### 5.2 停顿时间分析

| 恢复方式 | 停顿时间 | 有效训练时间 |
|----------|----------|--------------|
| **同步恢复** | ~10 分钟 | 44% |
| **FT-HSDP** | ~3 分钟 | 80% |

**停顿时间分解**：
- 故障检测超时：60 秒
- 功率平滑器：45 秒（已修复）
- FTAR 重配置 Bug：45 秒（已修复）
- 预期修复后：~1.5 分钟

**有效训练时间计算**：
```
同步恢复：(18-10)/18 = 44%
FT-HSDP：(8 + 7×11/12)/18 = 80%
```

### 5.3 吞吐量

| 状态 | 吞吐量 |
|------|--------|
| 12 副本正常运行 | 450 TFlops/GPU/s |
| 11 副本运行（故障恢复中） | 450 TFlops/GPU/s |
| 12 副本完全恢复 | 450 TFlops/GPU/s |

**关键发现**：FT-HSDP 在无故障情况下无额外开销。

### 5.4 模型精度影响

**实验设置**：
- 256 H100 GPU
- 3B 活跃参数 MoE 模型，16 专家
- 5000 亿文本 Token

**学习率策略对比**：

| 策略 | 说明 | 效果 |
|------|------|------|
| **none** | 基于步骤的学习率 | 有方差 |
| **linear** | 按健康副本比例缩放 | 减少方差 |
| **sqrt** | 平方根缩放 | 最佳，消除方差 |

**结论**：异步恢复不会显著影响最终模型质量。

## 六、核心技术详解

### 6.1 FTAR 协议细节

**CPU-GPU 交互流程**：

```
1. CPU 主线程初始化 RDMA 连接
2. CPU 运行 reconfig 确定 FTAR 组成员
3. 启动 GPU 流和 CPU 异步线程
4. GPU 流：复制数据到发送缓冲区，通知 CPU
5. CPU 异步线程：通过 RDMA 发送数据
6. CPU 异步线程：等待远程数据到达
7. 通知 GPU 流数据已到达
8. GPU 流：执行 reduce 操作
9. 重复直到所有梯度交换完成
```

**Ring 算法实现**：
- ReduceScatter 阶段：N-1 步
- AllGather 阶段：N-1 步
- 固定大小块流水线：控制并发数据包数量

**优化技术**：
- 合并存储操作减少 CPU-GPU 同步
- 利用寄存器避免 HBM 负载
- 指令级并行优化
- 低占用率高带宽内核设计

### 6.2 故障检测与处理

**故障分类**：

| 类型 | 处理方式 |
|------|----------|
| **可恢复错误** | 超时、网络错误 → 重试 |
| **不可恢复错误** | 硬件故障、NCCL 错误 → 崩溃 |

**超时机制**：
- 默认 60 秒超时
- 计划集成快速根因分析工具

### 6.3 一致性保证

**故障后一致性协议**：

```
检测到故障后：
  副本内运行类似两阶段提交(2PC)

  Rank 0 询问其他 Rank：
    - 所有 Rank 完成梯度交换？
      - 是：应用梯度，进入下一步
      - 否：丢弃梯度，重试当前步

  不同副本独立决策：
    - 可能处于不同步骤
    - 落后副本触发追赶协议
```

## 七、工程实践

### 7.1 内存压力缓解

**挑战**：更多小副本 → 更多状态存储

**解决方案**：
1. 测试不同副本数量找平衡点（10-20 副本）
2. 优化器状态卸载到 CPU 内存
   - 获取 1GB 状态：~60ms

### 7.2 减少首步效应

**问题**：恢复后首步包含耗时初始化

**优化**：
- 在检查前执行初始化函数
- 无法完全消除（如加载首批数据）

### 7.3 CPU 大规模仿真

**目的**：在 GPU 资源有限时测试大规模逻辑

**实现**：
- Shadow 模块加载机制
- GPU 模块替换为 CPU Mock 模块
- 跳过实际计算，传递张量形状
- 自定义 CPU 通信库替代 NCCL

**效果**：
- 检测并修复多个 Bug
- 例如：共识协议延迟从 9 秒优化到 700ms

### 7.4 快速根因分析

**目标**：快速定位故障 GPU/服务器

**方法**：
1. 构建集合通信等待图
2. 分析 GPU 级等待图
3. 聚合集合完成时间

**性能**：
- 每次分析 < 5 秒（32K+ GPU）
- 遥测数据 < 1MB/GPU
- 准确率 97.8%

## 八、相关工作

### 8.1 同步训练容错

| 方法 | 特点 | 局限 |
|------|------|------|
| **检查点重启** | 简单直接 | 恢复时间长 |
| **热备 GPU** | 减少分配时间 | 仍需全系统重启 |
| **RobustLLM** | 工程优化 | 规模受限 |

### 8.2 异步训练

| 方法 | 特点 | 局限 |
|------|------|------|
| **Dynamic SGD** | 异步梯度更新 | 收敛性问题 |
| **ReCycle** | 副本级恢复 | 规模有限 |
| **FT-HSDP** | 副本级容错 + 非阻塞追赶 | 首步效应 |

### 8.3 容错通信

| 方法 | 特点 | 局限 |
|------|------|------|
| **NCCL** | GPU 驱动，高性能 | 无法动态重配置 |
| **FTAR** | CPU-GPU 混合 | 需要自定义实现 |

## 九、总结

### 核心贡献

1. **FT-HSDP 范式**：数据并行副本作为容错单元
2. **FTAR 协议**：CPU 控制 + GPU 数据的混合设计
3. **非阻塞追赶**：利用训练特性实现零梯度同步
4. **工程实践**：100K GPU 规模验证

### 关键数据

| 指标 | 同步恢复 | FT-HSDP | 改善 |
|------|----------|---------|------|
| **停顿时间** | 10 分钟 | 3 分钟 | 3.3x |
| **有效训练时间** | 44% | 80% | 1.8x |
| **吞吐量开销** | - | 0% | - |

### 技术影响

FT-HSDP 证明了：
- 副本级容错是大规模训练的可行方案
- 异步恢复不会显著影响模型精度
- CPU-GPU 混合设计可实现高性能容错通信

### 局限性

1. **首步效应**：恢复后首步仍有额外延迟
2. **内存压力**：更多副本需要更多状态存储
3. **规模验证**：精度实验仅在 256 GPU 进行
4. **故障检测**：60 秒超时可能过长

### 未来方向

1. **快速故障检测**：集成根因分析工具
2. **动态副本数**：根据故障率调整副本数量
3. **跨 DC 优化**：减少跨数据中心通信开销
4. **硬件协同**：与 GPU 厂商合作改进容错能力
